# v0.2.8 Release Tutorial

This is a verification-oriented walkthrough for every user-facing item in `v0.2.8`.
Everything below was exercised on March 12, 2026.

For features that work directly from the CLI, this tutorial uses real CLI commands.
For features that need more app wiring than a standalone shell provides, this tutorial
uses the exact targeted `pytest` command that was run against the existing repo tests.

## Release Items Covered

- `fix: Force tool call on first turn in knowledge agent`
- `docs: citation tracing via thread get/trace`
- `docs: Redis connection setup guidance`
- `feat: Split RedisCluster from RedisInstance`
- `feat: pinned docs, skills, and support tickets`
- `feat: Allow Redis Enterprise cluster creation to use env admin defaults`
- `feat: Cluster-scoped diagnostics across CLI/API/UI/MCP`

## 2) Knowledge Agent First-Turn Tool Calls

This release forced the knowledge agent to call a tool on the first iteration instead
of answering from the system prompt alone.

Run:

```bash
uv run pytest \
  tests/unit/agent/test_knowledge_agent.py::TestKnowledgeAgentMethods::test_first_iteration_forces_tool_choice_required
```

This passed. It verifies that iteration `0` binds the model with
`tool_choice="required"`, which is the release behavior change.

## 4) RedisCluster / RedisInstance Split

The release introduced the cluster-first model: create a `RedisCluster`, then link
instances to it with `cluster_id`.

Create-path verification:

```bash
uv run pytest \
  tests/unit/cli/test_cli_clusters.py::test_cluster_create_redis_enterprise_success
```

This passed. It exercises `redis-sre-agent cluster create` for a Redis Enterprise
cluster.

Linking verification:

```bash
uv run pytest \
  tests/unit/cli/test_cli_instances.py::test_instance_create_accepts_cluster_id_when_cluster_exists_and_is_compatible
```

This passed. It exercises `redis-sre-agent instance create --cluster-id ...` and verifies
that the saved instance is linked to the cluster.

This is also the updated connection/setup flow described in the connection docs:
cluster first, instance second.

## 5) Redis Enterprise Admin Defaults From Environment

This release allowed cluster creation to fall back to environment defaults for
Redis Enterprise admin credentials.

Run:

```bash
uv run pytest \
  tests/unit/cli/test_cli_clusters.py::test_cluster_create_redis_enterprise_success_with_env_defaults
```

This passed. It verifies that `cluster create` resolves:

- `REDIS_ENTERPRISE_ADMIN_URL`
- `REDIS_ENTERPRISE_ADMIN_USERNAME`
- `REDIS_ENTERPRISE_ADMIN_PASSWORD`

when the CLI flags are omitted.

## 6) Cluster-Scoped Diagnostics In CLI

The CLI query path gained cluster-aware routing and validation.

Cluster context is forwarded to triage:

```bash
uv run pytest \
  tests/unit/cli/test_cli_query.py::test_query_with_cluster_uses_sre_agent_and_passes_cluster_context
```

Conflicting target IDs are rejected:

```bash
uv run pytest \
  tests/unit/cli/test_cli_query.py::test_query_rejects_both_instance_and_cluster_ids
```

Both passed. Together they verify:

- `redis-sre-agent query -c <cluster_id> ...` routes through cluster context
- the query layer rejects passing both an instance ID and a cluster ID

## 7) Cluster-Scoped Diagnostics In API

The task API now validates cluster context alongside instance context.

Run:

```bash
uv run pytest \
  tests/unit/api/test_tasks_api.py::TestTasksAPI::test_create_task_rejects_instance_and_cluster_together
```

This passed. It verifies the API rejects requests that try to send both identifiers
in the same task context.

## 8) Cluster-Scoped Diagnostics In MCP

The MCP server gained explicit `cluster_id` support for deep triage.

Run:

```bash
uv run pytest \
  tests/unit/mcp_server/test_mcp_server.py::TestDeepTriageTool::test_deep_triage_with_cluster_id
```

This passed. It verifies that `redis_sre_deep_triage(..., cluster_id="...")`
forwards cluster context into task creation.

## What To Do If You Want To Re-Run This Tutorial

1. Create the three temporary markdown files exactly as shown above.
2. Run `pipeline prepare-sources` against the temporary source directory.
3. Inspect `/tmp/redis-sre-agent-v028-tutorial-artifacts/2099-01-04/batch_manifest.json`
   and the generated JSON artifacts under `shared/`.
4. Run the targeted `pytest` commands for the remaining release items.

That gives you one short pass through the full `v0.2.8` feature set without adding
new checked-in fixtures or release-specific tests.
